# Loop Dashboard(loop-app）

フェーズ2・スライス2-1の成果物。「Xブックマーク→解説記事生成ループ」を題材にした、
ビルド不要の静的モバイルファーストダッシュボード。

背景・要件は `../フェーズ1/05_確定要件_フェーズ2計画.md` および
`../フェーズ1/01_ヒアリング要件.md` を参照。

## 起動方法

フレームワーク・ビルドツールは一切使わない(プレーンHTML/CSS/JS)。
ローカルで確認する場合はこのディレクトリ(`loop-app/`)を静的サーバーで配信するだけでよい。

```bash
cd loop-app
python3 -m http.server 8080
# ブラウザで http://localhost:8080 を開く
```

または:

```bash
cd loop-app
npx serve .
```

`fetch()` で `state/loop-state.json` を読み込むため、`file://` で直接
`index.html` を開くとブラウザのCORS制限で失敗する。必ずHTTPサーバー経由で開くこと。

## GitHub Pagesでの配信

このディレクトリはそのまま静的サイトとして配信できる。リポジトリ設定で
Pagesの公開元をこの`loop-app/`ディレクトリ(または`docs/`へのコピー)に向けるだけでよい。
ビルドステップは不要。

## ファイル構成

| ファイル | 役割 |
| --- | --- |
| `index.html` | ページ構造。モバイル1カラム/900px以上で2ペイン(左チャット・右ダッシュボード) |
| `styles.css` | スタイル。モバイルファースト、状態色はWCAG AA準拠を意識 |
| `app.js` | `loops.json` からループ一覧を読み、選択されたループの `loop-state.json` を fetch して各セクションを描画するプレーンJS |
| `loops.json` | ループレジストリ。どのループがどの状態ファイルを持つかの一覧(下記「ループの登録方法」参照) |
| `state/loop-state.json` | サンプル状態データ(手書き)。`?loop=sample` で表示される。スキーマは `STATE_SCHEMA.md` 参照 |
| `STATE_SCHEMA.md` | `loop-state.json` のフィールド仕様書 |

## ループの登録方法

このダッシュボードは複数のループ(`loops/<loop-id>/`)を切り替えて表示できる。
実データは各ループフォルダ配下の `state/loop-state.json` が正本であり、
`loop-app/` 自身は表示用のレジストリ(`loops.json`)しか持たない。

新しいループを追加する手順:

1. `loops/<新しいloop-id>/` を作り、そのループの `state/loop-state.json` を用意する
   (スキーマは `STATE_SCHEMA.md` 準拠)
2. `loop-app/loops.json` の `loops` 配列にエントリを追加する:

   ```json
   { "id": "<loop-id>", "name": "<画面に出す表示名>", "statePath": "../loops/<loop-id>/state/loop-state.json" }
   ```

   - `statePath` は `loop-app/index.html` を配信するドキュメント基準の相対パス
     (リポジトリルートで静的サーバーを立てて `/loop-app/` を開く運用を想定)
   - 既定で表示したいループを `defaultLoop` に設定する
   - そのループが `roadmap.json`(スキーマ: `loops/roadmap-loop/ROADMAP_SCHEMA.md`)を
     持つ場合は、任意フィールド `roadmapPath` を追加すると「ロードマップ」セクションが
     ゴールセクションの下・前回からの変化の上に描画される:

     ```json
     {
       "id": "<loop-id>",
       "name": "<画面に出す表示名>",
       "statePath": "../loops/<loop-id>/state/loop-state.json",
       "roadmapPath": "../loops/<loop-id>/roadmaps/<日付-スラッグ>/roadmap.json"
     }
     ```

     `roadmapPath` を指定しない、またはそのパスの `fetch` が失敗する
     (404・不正JSON・スキーマ不一致)場合は、ロードマップセクション自体が
     非表示になるだけで、他のセクションの描画には影響しない
     (`loop-app/app.js` の `loadRoadmap` はエラーを飲み込む設計)

3. ブラウザで `?loop=<loop-id>` を付けて開けば、そのループの状態を確認できる。
   クエリなしでアクセスすると `defaultLoop` が使われる
4. ループが1件しかない場合は切替UI(`<select>`)は自動的に非表示になる。
   2件以上登録するとヘッダー下に切替UIが出る
5. `loops.json` 自体が存在しない/壊れている場合は、後方互換として
   従来どおり `./state/loop-state.json` を直接読みに行く(切替UIなし)

### 動作確認上の注意

`loop-app/` 単体を静的サーバーで配信すると `../loops/...` へは辿れない
(ドキュメントルートの外に出るパスになるため)。ループ間切り替えを含めて
確認する場合は、リポジトリルートで `python3 -m http.server` を起動し、
`http://localhost:8080/loop-app/` を開くこと(`loop-app/` 単体配信は
`?loop=sample` のような同一ディレクトリ内ループの確認にのみ有効)。

## 設計意図

このアプリの存在理由は機能の多さではなく **認知負荷の削減** にある
(`フェーズ1/05_確定要件_フェーズ2計画.md` より)。評価基準は一貫して:

- 文章を読まずに、見て分かるか
- 前回との差分が一目で分かるか

この2点を満たすため、以下の判断をした。

### 固定構造・自由レイアウトを避ける

チャットの自由形式ではなく、ゴール→差分→カンバン→ボトルネック→エラー→
スキル/教訓→おすすめ、という**固定の縦順序**にした。ループという反復構造には
毎回同じ場所に同じ種類の情報がある方が「見て分かる」。

### 差分セクションをカンバンより上に置く

前回からの変化だけを抜き出した `changes` を、カンバン全体より先(上)に表示する。
ユーザーが最初に見るべきは「全体像」ではなく「前回から何が変わったか」だと判断した
(要件書: 「前とどこが変わったか」を自力で突き合わせる必要がある、という課題認識に対応)。

### details/summaryで「壁」を作らない

タスクカード・エラーカード・停止条件は初期状態では1〜2行の要約のみを表示し、
詳細は `<details>` のネイティブ開閉に任せた。JSでのモーダルやアコーディオン実装を
避けることで、依存ライブラリなしでもアクセシブルな展開UIになる。

### モバイルファースト+900pxで2ペイン

要件（「ユーザーはスマホのみの生活」）を最優先し、375px幅を基準にレイアウトを作った上で、
900px以上のときだけ左にチャット列（現状はプレースホルダー）、右にダッシュボードの
2ペインに拡張する。将来チャット入力を接続する際も、右側のダッシュボード構造は変えずに済む。

### カンバンだけ横スクロールを許容する

ページ全体は縦スクロールのみとし、カンバンの5列(未着手/実行中/検証中/完了/エラー)
だけを横スクロールコンテナにした。5列を無理に縮めて可読性を落とすより、
「ここだけ横にスワイプする」という一貫した操作を許容する方がスマホでは自然と判断した。

### 状態色とコントラスト

`done`=緑、`doing`=青、`error`=赤、`review`=琥珀、`todo`=グレー。
いずれも「薄い背景色 + 濃い文字色」の組み合わせにし、WCAG AA(通常文字4.5:1以上)を
満たす配色にしている。全体のベースはニュートラルなライトグレー系で、絵文字は使用していない。

## 既知の制約(スライス2-1時点)

- `loop-state.json` は手書きのサンプルであり、実データではない(実データ連携はスライス2-2)
- 差分ハイライトの自動検出ロジック・ボトルネック検出の閾値計算はまだ実装しない
  (スライス2-3で対応予定。今回は静的サンプルデータを人手で作成している)
- チャット列は表示のみのプレースホルダー
